================================================================================
    DOMAIN MODEL DESCRIPTION - END-TO-END DELIVERY SYSTEM
    Detailed Documentation of All Classes, Attributes, and Relationships
================================================================================

OVERVIEW:
---------
The domain model represents the core business entities and their relationships
in the End-to-End Delivery Management System. It follows object-oriented 
principles with inheritance hierarchies, associations, and multiplicities.

================================================================================
SECTION 1: USER HIERARCHY (INHERITANCE STRUCTURE)
================================================================================

1.1 USER (ABSTRACT BASE CLASS)
------------------------------
Type: Abstract Class
Color Code: Light Blue (#dae8fc)
Purpose: Base class for all system users, providing common authentication 
         and account management functionality

Attributes:
- user_id: String
  * Unique identifier for each user
  * Primary key for user identification
  
- username: String
  * Unique login name
  * Used for authentication
  
- password: String
  * Hashed password for security
  * Stored using bcrypt/argon2 encryption
  
- email: String
  * User's email address
  * Used for notifications and password recovery
  
- phone: String
  * Contact phone number
  * Used for SMS notifications
  
- account_status: String
  * Current account state (Active, Locked, Suspended, Inactive)
  * Controls user access to system
  
- created_date: Date
  * Account creation timestamp
  * Used for auditing and analytics
  
- last_login: Date
  * Most recent login timestamp
  * Used for security monitoring

Relationships:
- Specialized by: Customer, Manager, Dispatcher, Rider, Admin (inheritance)


1.2 CUSTOMER (INHERITS FROM USER)
----------------------------------
Type: Concrete Class
Color Code: Light Green (#d5e8d4)
Purpose: Represents customers who place delivery orders

Attributes:
- delivery_address: String
  * Default delivery location
  * Can be overridden per order
  
- billing_address: String
  * Billing/invoice address
  * May differ from delivery address
  
- customer_type: String
  * Customer category (Individual, Business, Premium, Regular)
  * Affects pricing and priority
  
- loyalty_points: Integer
  * Accumulated reward points
  * Used for discounts and promotions

Inherited Attributes:
- All attributes from User class

Relationships:
- places → Order (1 to *)
  * One customer can place multiple orders
  * Each order belongs to exactly one customer
  
- receives → Notification (1 to *)
  * Customer receives multiple notifications
  * Each notification is sent to one customer


1.3 MANAGER (INHERITS FROM USER)
---------------------------------
Type: Concrete Class
Color Code: Light Green (#d5e8d4)
Purpose: Manages order approval and dispatcher delegation

Attributes:
- department: String
  * Manager's organizational department
  * Example: Operations, Logistics, Customer Service
  
- authorization_level: Integer
  * Approval authority level (1-10)
  * Determines what orders can be approved
  
- assigned_region: String
  * Geographic region managed
  * Example: North Zone, Central Region

Inherited Attributes:
- All attributes from User class

Relationships:
- approves → Order (1 to *)
  * Manager approves multiple orders
  * Each order approved by one manager
  
- delegates to → Dispatcher (many to many implied)
  * Manager assigns orders to dispatchers
  
- generates → Report (1 to *)
  * Manager creates various reports
  * For analytics and decision making


1.4 DISPATCHER (INHERITS FROM USER)
------------------------------------
Type: Concrete Class
Color Code: Light Green (#d5e8d4)
Purpose: Coordinates pickup and delivery assignments

Attributes:
- assigned_zone: String
  * Geographic zone for operations
  * Determines which orders can be assigned
  
- availability_status: String
  * Current working status (Available, Busy, Off-duty)
  * Affects assignment capability
  
- active_orders_count: Integer
  * Number of currently managed orders
  * Used for workload balancing

Inherited Attributes:
- All attributes from User class

Relationships:
- creates → Assignment (1 to *)
  * Dispatcher creates multiple assignments
  * Links riders to orders
  
- receives delegation from → Manager
  * Manager delegates orders to dispatcher


1.5 RIDER (INHERITS FROM USER)
-------------------------------
Type: Concrete Class
Color Code: Light Green (#d5e8d4)
Purpose: Pickup riders and delivery riders who handle parcels

Attributes:
- vehicle_type: String
  * Type of transport (Bike, Car, Van, Motorcycle)
  * Determines capacity and parcel size limits
  
- license_number: String
  * Vehicle registration or driver's license
  * Required for legal compliance
  
- availability_status: String
  * Current status (Available, On-delivery, Off-duty, Break)
  * Real-time availability for assignments
  
- current_location: String
  * GPS coordinates or address
  * Used for proximity-based assignment
  
- rider_type: String
  * Role type (Pickup Rider, Delivery Rider, Both)
  * Determines assignment eligibility
  
- rating: Float
  * Performance rating (0.0 to 5.0)
  * Based on customer feedback and metrics

Inherited Attributes:
- All attributes from User class

Relationships:
- receives → Assignment (1 to *)
  * Rider receives multiple assignments
  * Each assignment goes to one rider
  
- follows → Route (1 to *)
  * Rider follows multiple routes
  * Each route assigned to one rider


1.6 ADMIN (INHERITS FROM USER)
-------------------------------
Type: Concrete Class
Color Code: Light Green (#d5e8d4)
Purpose: System administration and reporting

Attributes:
- privileges: String[]
  * Array of admin permissions
  * Examples: user_management, system_config, data_access
  
- access_level: String
  * Admin tier (Super Admin, Admin, Support)
  * Determines scope of access
  
- department: String
  * Administrative department
  * Example: IT, Operations, Finance

Inherited Attributes:
- All attributes from User class

Relationships:
- generates → Report (1 to *)
  * Admin creates system-wide reports
  * Includes performance, audit, and compliance reports

SECTION 2: CORE BUSINESS ENTITIES

2.1 ORDER (CENTRAL ENTITY)
---------------------------
Type: Concrete Class
Color Code: Yellow (#fff2cc)
Purpose: Represents a delivery order throughout its lifecycle

Attributes:
- order_id: String
  * Unique order identifier
  * Primary key, visible to customers
  
- customer_id: String
  * Foreign key to Customer
  * Links order to customer who placed it
  
- status: String
  * Current order state
  * Values: Created, Pending Approval, Paid, Approved, Assigned to Dispatcher,
    Pickup Assigned, In Warehouse, Out for Delivery, Delivered, Cancelled
  
- order_date: Date
  * Order creation timestamp
  * Start of order lifecycle
  
- approved_date: Date
  * Manager approval timestamp
  * Null if not yet approved
  
- pickup_address: String
  * Location to collect parcel
  * Complete address with coordinates
  
- delivery_address: String
  * Final destination address
  * Complete address with coordinates
  
- estimated_delivery: Date
  * Predicted delivery date/time
  * Based on distance and current load
  
- actual_delivery: Date
  * Actual completion timestamp
  * Null until delivered
  
- priority: String
  * Order urgency (Standard, Express, Same-day)
  * Affects cost and routing
  
- total_cost: Float
  * Total order amount
  * Includes delivery charges and taxes
  
- special_instructions: String
  * Customer notes
  * Example: Fragile, Handle with care, Gate code
  
- approved_by: String
  * Manager ID who approved order
  * For audit trail

Relationships:
- placed by → Customer (1 to 1)
  * Each order placed by one customer
  
- approved by → Manager (1 to 1)
  * Each order approved by one manager
  
- has → Payment (1 to 1)
  * One payment per order
  * Payment must exist before approval
  
- has → ShipmentPicture (1 to *)
  * Order can have multiple pictures
  * At least one picture required
  
- contains → Parcel (1 to 1..*)
  * Order contains one or more parcels
  * Multiple parcels for large shipments
  
- has → Assignment (1 to *)
  * Multiple assignments throughout lifecycle
  * Pickup assignment, delivery assignment
  
- has → Route (1 to 0..1)
  * Optional route when out for delivery
  * Created when delivery rider assigned
  
- triggers → Notification (1 to *)
  * Status changes trigger notifications
  * Multiple notifications per order
  
- has → Tracking (1 to *)
  * Multiple tracking points
  * Real-time location updates
  
- tracks → StatusHistory (1 to *)
  * Complete audit trail of status changes
  * One entry per status transition


2.2 PAYMENT
-----------
Type: Concrete Class
Color Code: Light Orange (#ffe6cc)
Purpose: Records payment transactions for orders

Attributes:
- payment_id: String
  * Unique payment identifier
  * Primary key
  
- order_id: String
  * Foreign key to Order
  * Links payment to specific order
  
- amount: Float
  * Payment amount in system currency
  * Matches order total_cost
  
- payment_method: String
  * Payment type (Credit Card, Debit Card, PayPal, Bank Transfer, Digital Wallet)
  * Customer's chosen method
  
- payment_date: Date
  * Transaction timestamp
  * When payment processed
  
- status: String
  * Payment state (Pending, Paid, Failed, Refunded, Cancelled)
  * Critical for order approval
  
- transaction_id: String
  * Gateway transaction reference
  * For reconciliation and disputes
  
- gateway_response: String
  * Response from payment gateway
  * Success/failure message from processor
  
- refund_amount: Float
  * Amount refunded if applicable
  * 0.0 if no refund issued

Relationships:
- has → Order (1 to 1 bidirectional)
  * Each payment links to exactly one order
  * Each order has exactly one payment


2.3 SHIPMENT PICTURE
--------------------
Type: Concrete Class
Color Code: Light Purple (#e1d5e7)
Purpose: Stores images of shipments for verification

Attributes:
- picture_id: String
  * Unique picture identifier
  * Primary key
  
- order_id: String
  * Foreign key to Order
  * Links picture to order
  
- image_url: String
  * Storage location (cloud URL or file path)
  * Points to actual image file
  
- upload_date: Date
  * Upload timestamp
  * When picture was added
  
- file_size: Integer
  * Size in bytes
  * For storage management
  
- format: String
  * Image format (JPG, PNG, HEIC)
  * Must be supported format
  
- uploaded_by: String
  * User ID who uploaded
  * Customer or staff member

Relationships:
- has → Order (many to 1)
  * Multiple pictures can belong to one order
  * Each picture belongs to exactly one order


2.4 PARCEL
----------
Type: Concrete Class
Color Code: Light Red (#f8cecc)
Purpose: Physical item being delivered

Attributes:
- parcel_id: String
  * Unique parcel identifier
  * Primary key, barcode/QR code
  
- order_id: String
  * Foreign key to Order
  * Links parcel to order
  
- weight: Float
  * Weight in kilograms
  * Affects cost and vehicle assignment
  
- dimensions: String
  * Size (length x width x height in cm)
  * Format: "30x20x10"
  
- status: String
  * Parcel state (Created, Collected, In Warehouse, In Transit, Delivered)
  * Independent tracking from order
  
- description: String
  * Item description
  * What's being delivered
  
- fragile: Boolean
  * Indicates if special handling required
  * Affects packaging and routing
  
- collected_date: Date
  * Pickup timestamp
  * When rider collected from customer
  
- delivered_date: Date
  * Delivery completion timestamp
  * When delivered to recipient

Relationships:
- contains → Order (many to 1)
  * Multiple parcels can belong to one order
  * Each parcel belongs to exactly one order
  
- stored in → Inventory (1 to 0..1)
  * Parcel may be in warehouse inventory
  * Optional, only when in warehouse

Relationships:

  * Multiple inventory entries per warehouse

Warehouse Relationships:
- Warehouse manages Inventory (1:*)

Admin Relationships:
- Admin generates Report (1:*)


5.3 MULTIPLICITIES EXPLANATION
-------------------------------

Notation: (min..max)
- 1 = exactly one (mandatory)
- * = zero or many
- 0..1 = optional (zero or one)
- 1..* = one or many (at least one required)

Examples:
- Customer places Order (1:*)
  * One customer can place zero or many orders
  * Each order must be placed by exactly one customer

- Order contains Parcel (1:1..*)
  * One order contains at least one parcel (could be multiple)
  * Each parcel belongs to exactly one order

- Parcel stored in Inventory (1:0..1)
  * One parcel may or may not be in inventory (optional)
  * When in inventory, stored in exactly one warehouse

SECTION 6: DOMAIN MODEL DESIGN PRINCIPLES
================================================================================

6.1 NORMALIZATION
-----------------
- Each entity has a unique identifier (primary key)
- Foreign keys establish relationships
- No redundant data storage
- Attributes are atomic (single-valued)

6.2 ABSTRACTION
---------------
- User is abstract base class
- Concrete user types inherit common attributes
- Promotes code reuse and consistency

6.3 ENCAPSULATION
-----------------
- Each entity encapsulates related attributes
- Responsibilities are clear and separated
- Order is central entity coordinating workflow

6.4 BUSINESS RULES ENCODED
---------------------------
- Payment must exist before order approval (1:1 mandatory)
- Order must have at least one parcel (1..*)
- Parcel only in inventory while in warehouse (0..1 optional)
- Multiple status changes tracked (1:* history)

6.5 SCALABILITY CONSIDERATIONS
-------------------------------
- Assignment entity allows flexible dispatcher-rider-order links
- Tracking allows unlimited location points
- StatusHistory provides complete audit trail
- Notification supports multi-channel delivery

SECTION 8: TYPICAL WORKFLOW THROUGH DOMAIN MODEL
================================================================================

Order Lifecycle Example:
------------------------

1. Customer (User) places Order
   - Creates Order entity
   - Uploads ShipmentPicture
   - Order status: "Created"

2. Customer makes Payment
   - Creates Payment entity
   - Links to Order
   - Order status: "Paid"

3. Manager (User) reviews Order
   - Checks Payment status
   - Reviews ShipmentPicture
   - Approves Order
   - Order.approved_by = manager_id
   - Order status: "Approved"
   - Creates StatusHistory record

4. Manager delegates to Dispatcher
   - Order status: "Assigned to Dispatcher"

5. Dispatcher (User) creates Assignment
   - Links Order to Rider (pickup)
   - Assignment type: "Pickup"
   - Order status: "Pickup Assigned"

6. Rider (User) collects Parcel
   - Updates Parcel status: "Collected"
   - Creates Tracking points
   - Order status: "In Warehouse"

7. Warehouse receives Parcel
   - Creates Inventory entry
   - Records location_in_warehouse
   - Parcel.status: "In Warehouse"

8. Warehouse dispatches for delivery
   - Creates new Assignment (delivery rider)
   - Creates Route entity
   - Updates Inventory.exit_date
   - Order status: "Out for Delivery"
   - Triggers Notification to Customer

9. Delivery Rider follows Route
   - Updates Tracking with GPS coordinates
   - Customer can view real-time location

10. Delivery completed
    - Order status: "Delivered"
    - Parcel.delivered_date = now
    - Creates final StatusHistory entry
    - Triggers final Notification

11. Manager/Admin generates Report
    - Analyzes completed orders
    - Creates Report entity
    - Reviews performance metrics
