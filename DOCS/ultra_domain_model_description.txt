SECTION 1: OVERVIEW AND DESIGN PHILOSOPHY
================================================================================

Purpose:
--------
This ultra domain model represents the complete conceptual design for an
End-to-End Delivery Management System. It follows pure domain-driven design
principles, focusing on business concepts rather than implementation details.

Key Principles Applied:
-----------------------
✓ Pure domain model (no technical implementation details)
✓ Minimal but meaningful attributes (only those critical for relationships)
✓ Clear separation of concerns
✓ Proper use of inheritance for "is-a" relationships
✓ Associations for "has-a" and "uses" relationships
✓ Dependencies shown for behavioral relationships
✓ No crossing lines in visual layout
✓ Beautiful and intuitive organization

Entities Count: 18 classes
Relationships: 25+ associations
Inheritance Depth: 2 levels (User → specific user types)

SECTION 2: ISSUES FIXED FROM BASIC MODEL
================================================================================

Critical Fixes Applied:
-----------------------

1. SEPARATED RIDER TYPES
   Before: Single "Rider" class with rider_type attribute
   After: PickupRider and DeliveryRider as separate classes
   Reason: Different responsibilities, different workflows, different vehicle needs

2. SPLIT ASSIGNMENT ENTITY
   Before: Single Assignment with assignment_type attribute
   After: PickupAssignment and DeliveryAssignment as separate classes
   Reason: Different lifecycles, different actors, clearer domain model

3. ADDED WAREHOUSE STAFF
   Before: Missing from model but present in use cases
   After: WarehouseStaff as distinct User subclass
   Reason: Critical actor in UC08, has unique responsibilities

4. EXTRACTED ADDRESS ENTITY
   Before: Addresses stored as strings in multiple places
   After: Address as separate entity with reusable relationships
   Reason: Eliminates duplication, enables address validation and geocoding

5. ADDED ZONE ENTITY
   Before: Zone mentioned in attributes but not modeled
   After: Zone as first-class entity
   Reason: Critical for dispatcher/rider assignment and routing

6. ADDED VEHICLE ENTITY
   Before: Vehicle attributes mixed into Rider
   After: Vehicle as separate entity
   Reason: Riders can change vehicles, vehicle has its own lifecycle

7. ADDED POLICY RULE ENTITY
   Before: Policy checking mentioned but not modeled
   After: PolicyRule as entity
   Reason: Policies are configurable business rules, not hardcoded

8. REMOVED TECHNICAL ATTRIBUTES
   Before: Many ID fields, timestamps, URLs
   After: Only domain-relevant attributes
   Reason: Domain model focuses on "what" not "how"

9. IMPROVED RELATIONSHIPS
   Before: Generic associations
   After: Specific, named relationships with clear semantics
   Reason: Better understanding of business rules

10. BETTER VISUAL LAYOUT
    Before: Lines crossing, cluttered
    After: Clean zones, no crossing lines, grouped by concern
    Reason: Easier to understand and present

3.1 USER HIERARCHY (7 ENTITIES)
--------------------------------

USER (ABSTRACT BASE CLASS)
--------------------------
Purpose: Foundation for all system users
Type: Abstract class (cannot be instantiated)
Color: Light Blue (#dae8fc)

Attributes:
- username: String
  * Unique login identifier
  * Used for authentication
  
- password: String
  * Encrypted credential
  * Stored as hash (bcrypt/argon2)
  
- email: String
  * Contact and notification
  * Unique per user
  
- accountStatus: String
  * Domain values: Active, Locked, Suspended, Inactive
  * Controls system access

Specializations: Customer, Manager, Dispatcher, PickupRider, DeliveryRider, 
                 WarehouseStaff, Admin


CUSTOMER
--------
Purpose: End user who places delivery orders
Type: Concrete class
Color: Light Green (#d5e8d4)
Inherits from: User

Specific Attributes:
- customerType: String
  * Domain values: Individual, Business, Premium, Regular
  * Affects pricing and service level
  
- loyaltyPoints: Integer
  * Accumulated rewards
  * Used for discounts

Relationships:
- places → Order (1 to many)
  * Customer places multiple orders over time
  
- receives → Notification (1 to many)
  * Receives status updates and alerts


MANAGER
-------
Purpose: Approves orders and delegates to dispatchers
Type: Concrete class
Color: Light Green (#d5e8d4)
Inherits from: User

Specific Attributes:
- authorizationLevel: Integer
  * Approval authority (1-10)
  * Higher levels can approve high-value orders
  
- department: String
  * Organizational unit
  * Examples: Operations, Logistics, Customer Service

Relationships:
- approves → Order (1 to many) [dependency]
  * Manager approves orders before processing
  
- delegates to → Dispatcher (many to many) [dependency]
  * Assigns approved orders to dispatchers
  
- generates → Report (1 to many) [dependency]
  * Creates performance and analytics reports


DISPATCHER
----------
Purpose: Coordinates pickup and delivery assignments
Type: Concrete class
Color: Light Green (#d5e8d4)
Inherits from: User

Specific Attributes:
- availabilityStatus: String
  * Domain values: Available, Busy, Off-duty
  * Determines assignment eligibility
  
- currentLoad: Integer
  * Number of active assignments
  * Used for load balancing

Relationships:
- creates → PickupAssignment (1 to many) [dependency]
  * Assigns pickup riders to orders
  
- operates in → Zone (many to 1) [dependency]
  * Geographic area of responsibility


PICKUP RIDER

Color: Light Green (#d5e8d4)

Specific Attributes:

3.2 CORE BUSINESS ENTITIES (4 ENTITIES)
----------------------------------------

ORDER
-----
Purpose: Central entity representing delivery request
Type: Concrete class
Color: Yellow (#fff2cc) - EMPHASIZED with thick border
Importance: ★★★★★ Core entity around which entire system revolves

Attributes:
- orderNumber: String
  * Human-readable order identifier
  * Visible to customers (e.g., "ORD-2025-001234")
  
- status: String
  * Current state in order lifecycle
  * Domain values: Created, Pending Approval, Paid, Approved, 
    Assigned to Dispatcher, Pickup Assigned, In Warehouse, 
    Out for Delivery, Delivered, Cancelled, Rejected
  
- orderDate: Date
  * When order was created
  * Start of lifecycle
  
- priority: String
  * Urgency level
  * Domain values: Standard, Express, Same-day
  * Affects cost and routing
  
- totalCost: Float
  * Total charge including delivery
  * Calculated based on weight, distance, priority
  
- specialInstructions: String
  * Customer notes
  * Examples: "Fragile", "Leave at door", "Call on arrival"

Relationships:
- placed by → Customer (many to 1)
  * Every order must have a customer
  * Customer can place many orders
  
- paid via → Payment (1 to 1)
  * One payment per order
  * Payment mandatory before approval
  
- contains → Parcel (1 to 1..many)
  * Order has at least one parcel
  * Can contain multiple parcels
  
- has → ShipmentPicture (1 to 1..many)
  * At least one picture required
  * Multiple pictures allowed for documentation
  
- pickup from → Address (1 to 1)
  * Pickup location
  * Where parcel is collected
  
- deliver to → Address (1 to 1)
  * Delivery destination
  * Final parcel destination
  
- has → PickupAssignment (1 to 0..1)
  * Optional, exists after dispatcher assigns pickup rider
  
- has → DeliveryAssignment (1 to 0..1)
  * Optional, exists after warehouse assigns delivery rider
  
- triggers → Notification (1 to many)
  * Status changes trigger notifications
  
- tracks → StatusHistory (1 to many)
  * Complete audit trail of status changes
  
- approved by → Manager (many to 1) [dependency]
  * Manager approval required
  
- validated by → PolicyRule (many to many) [dependency]
  * Checked against business policies


PAYMENT
-------
Purpose: Financial transaction for order
Type: Concrete class
Color: Light Orange (#ffe6cc)
Importance: ★★★★ Critical for order processing

Attributes:
- amount: Float
  * Payment value
  * Matches Order.totalCost
  
- paymentMethod: String
  * How customer paid
  * Domain values: Credit Card, Debit Card, PayPal, Bank Transfer, 
    Digital Wallet, Cash on Delivery
  
- status: String
  * Transaction state
  * Domain values: Pending, Processing, Paid, Failed, Refunded, Cancelled
  
- transactionRef: String
  * Gateway reference number
  * For reconciliation and dispute resolution

Relationships:
- paid via → Order (1 to 1 bidirectional)
  * Each payment for exactly one order
  * Each order has exactly one payment


PARCEL
------
Purpose: Physical item being transported
Type: Concrete class
Color: Light Red (#f8cecc)
Importance: ★★★★ Physical manifestation of order

Attributes:
- weight: Float
  * Weight in kilograms
  * Affects cost and vehicle assignment
  
- dimensions: String
  * Size specification
  * Format: "length x width x height" (cm)
  * Affects capacity planning
  
- description: String
  * What's being delivered
  * Content description
  
- fragile: Boolean
  * Special handling flag
  * Affects packaging and routing
  
- status: String
  * Parcel lifecycle state
  * Domain values: Created, Collected, In Warehouse, In Transit, 
    Delivered, Damaged, Lost

Relationships:
- contains → Order (many to 1)
  * Parcel belongs to exactly one order
  * Order can have multiple parcels
  
- stored in → Inventory (1 to 0..1)
  * Parcel may be in warehouse inventory
  * Optional, only when physically in warehouse


SHIPMENT PICTURE
----------------
Purpose: Visual documentation of shipment
Type: Concrete class
Color: Light Purple (#e1d5e7)
Importance: ★★★ Required for manager approval

Attributes:
- imageUrl: String
  * Storage location reference
  * Points to cloud storage or file system
  
- uploadDate: Date
  * When picture was uploaded
  * Part of order timeline
  
- format: String
  * Image file type
  * Domain values: JPG, PNG, HEIC
  * Must be supported format

Relationships:
- has → Order (many to 1)
  * Picture belongs to exactly one order
  * Order requires at least one picture

SECTION 4: INHERITANCE HIERARCHY
================================================================================

Complete Inheritance Tree:
--------------------------

User (abstract)
│
├── Customer
│   └── Places orders, receives notifications
│
├── Manager  
│   └── Approves orders, delegates work, generates reports
│
├── Dispatcher
│   └── Creates pickup assignments, operates in zones
│
├── PickupRider
│   └── Collects parcels, uses vehicle
│
├── DeliveryRider
│   └── Delivers parcels, uses vehicle, follows routes
│
├── WarehouseStaff
│   └── Works at warehouse, manages inventory
│
└── Admin
    └── System administration, generates reports


Inheritance Benefits:
---------------------
1. CODE REUSE
   - All user types share common authentication attributes
   - username, password, email, accountStatus inherited by all

2. POLYMORPHISM
   - Any user type can authenticate
   - Different user types have different dashboards
   - Role-based access control simplified

3. EXTENSIBILITY
   - New user types can be added easily
   - Example: Could add "Customer Support" or "Finance Officer"

4. CLARITY
   - Clear "is-a" relationships
   - PickupRider IS-A User
   - DeliveryRider IS-A User (not just Rider with type attribute)

SECTION 5: ASSOCIATION RELATIONSHIPS (DETAILED)
================================================================================

5.1 ASSOCIATIONS (SOLID LINES)
-------------------------------
Represent structural "has-a" or "part-of" relationships
These are permanent, data-driven connections

1. Customer places Order (1 : *)
   - One customer places zero or many orders
   - Each order placed by exactly one customer
   - Multiplicity: Customer[1] ←→ Order[*]

2. Order paid via Payment (1 : 1)
   - Each order has exactly one payment
   - Each payment belongs to exactly one order
   - Multiplicity: Order[1] ←→ Payment[1]

3. Order contains Parcel (1 : 1..*)
   - Each order contains at least one parcel
   - Each parcel belongs to exactly one order
   - Multiplicity: Order[1] ←→ Parcel[1..*]

4. Order has ShipmentPicture (1 : 1..*)
   - Each order has at least one picture
   - Each picture belongs to exactly one order
   - Multiplicity: Order[1] ←→ ShipmentPicture[1..*]

5. Order pickup from Address (1 : 1)
   - Each order has exactly one pickup address
   - Address can be pickup for many orders
   - Multiplicity: Order[1] ←→ Address[1]

6. Order deliver to Address (1 : 1)
   - Each order has exactly one delivery address
   - Address can be delivery for many orders
   - Multiplicity: Order[1] ←→ Address[1]

7. Order has PickupAssignment (1 : 0..1)
   - Order may have zero or one pickup assignment
   - Assignment belongs to exactly one order
   - Multiplicity: Order[1] ←→ PickupAssignment[0..1]

8. Order has DeliveryAssignment (1 : 0..1)
   - Order may have zero or one delivery assignment
   - Assignment belongs to exactly one order
   - Multiplicity: Order[1] ←→ DeliveryAssignment[0..1]

9. PickupAssignment assigned to PickupRider (* : 1)
   - Assignment assigned to exactly one pickup rider
   - Rider can have multiple assignments
   - Multiplicity: PickupAssignment[*] ←→ PickupRider[1]

10. DeliveryAssignment assigned to DeliveryRider (* : 1)
    - Assignment assigned to exactly one delivery rider
    - Rider can have multiple assignments
    - Multiplicity: DeliveryAssignment[*] ←→ DeliveryRider[1]

11. DeliveryAssignment follows Route (1 : 1)
    - Each delivery assignment has exactly one route
    - Each route belongs to one delivery assignment
    - Multiplicity: DeliveryAssignment[1] ←→ Route[1]

12. Parcel stored in Inventory (1 : 0..1)
    - Parcel may or may not be in inventory
    - When in inventory, one inventory entry per parcel
    - Multiplicity: Parcel[1] ←→ Inventory[0..1]

13. Warehouse manages Inventory (1 : *)
    - Warehouse manages multiple inventory entries
    - Each inventory entry in one warehouse
    - Multiplicity: Warehouse[1] ←→ Inventory[*]

14. PickupRider uses Vehicle (* : 1)
    - Rider uses one vehicle at a time
    - Vehicle can be used by multiple riders (shifts)
    - Multiplicity: PickupRider[*] ←→ Vehicle[1]

15. DeliveryRider uses Vehicle (* : 1)
    - Rider uses one vehicle at a time
    - Vehicle can be used by multiple riders (shifts)
    - Multiplicity: DeliveryRider[*] ←→ Vehicle[1]

16. Zone covers Address (1 : *)
    - Zone covers multiple addresses
    - Each address in exactly one zone
    - Multiplicity: Zone[1] ←→ Address[*]

17. Route tracked by Tracking (1 : *)
    - Route has multiple tracking points
    - Each tracking point belongs to one route
    - Multiplicity: Route[1] ←→ Tracking[*]

18. Customer receives Notification (1 : *)
    - Customer receives multiple notifications
    - Each notification sent to one customer
    - Multiplicity: Customer[1] ←→ Notification[*]

19. Order triggers Notification (1 : *)
    - Order triggers multiple notifications
    - Each notification relates to one order
    - Multiplicity: Order[1] ←→ Notification[*]

20. Order tracks StatusHistory (1 : *)
    - Order has multiple status changes
    - Each history entry for one order
    - Multiplicity: Order[1] ←→ StatusHistory[*]


5.2 DEPENDENCIES (DASHED LINES)
--------------------------------
Represent behavioral "uses" or "depends-on" relationships
These are action-oriented, not structural

1. Manager approves Order (1 : *)
   - Manager performs approval action
   - Not stored as permanent link, but recorded in audit

2. Manager delegates to Dispatcher (many : many)
   - Manager assigns orders to dispatchers
   - Action-based relationship

3. Dispatcher creates PickupAssignment (1 : *)
   - Dispatcher creates assignment as action
   - Assignment then becomes structural

4. Dispatcher operates in Zone (* : 1)
   - Dispatcher assigned to work in zone
   - Determines which orders they handle

5. WarehouseStaff works at Warehouse (* : 1)
   - Staff assigned to warehouse
   - Determines their scope of work

6. Manager generates Report (1 : *)
   - Manager creates reports
   - Action that produces Report entity

7. Admin generates Report (1 : *)
   - Admin creates system reports
   - Broader scope than manager reports

8. Order validated by PolicyRule (* : *)
   - Orders checked against policies
   - Policies apply to many orders

SECTION 6: BUSINESS RULES AND CONSTRAINTS
================================================================================

6.1 MANDATORY RELATIONSHIPS
----------------------------
These relationships MUST exist for valid business state:

1. Order MUST have exactly one Customer
   - Cannot create order without customer
   
2. Order MUST have exactly one Payment (before approval)
   - Payment required for manager approval
   
3. Order MUST contain at least one Parcel
   - Cannot have empty order
   
4. Order MUST have at least one ShipmentPicture
   - Picture required for manager verification
   
5. Order MUST have pickup and delivery Addresses
   - Both required for logistics
   
6. DeliveryAssignment MUST have Route
   - Route created automatically with delivery assignment
   
7. Warehouse MUST manage at least one Inventory entry (operational)
   - Empty warehouse has no purpose


6.2 OPTIONAL RELATIONSHIPS
---------------------------
These may or may not exist depending on order state:

1. Order may have 0 or 1 PickupAssignment
   - Created only after dispatcher assigns pickup rider
   
2. Order may have 0 or 1 DeliveryAssignment
   - Created only after warehouse assigns delivery rider
   
3. Parcel may have 0 or 1 Inventory entry
   - Only when parcel is in warehouse
   - Before pickup or after delivery: no inventory


6.3 STATE-DEPENDENT CONSTRAINTS
--------------------------------

Order Status Constraints:
- "Created" → Payment must exist
- "Paid" → Payment.status must be "Paid"
- "Approved" → Manager approval recorded
- "Pickup Assigned" → PickupAssignment must exist
- "In Warehouse" → Inventory entry must exist
- "Out for Delivery" → DeliveryAssignment and Route must exist
- "Delivered" → Route.actualTime must be set

Parcel Status Alignment:
- Parcel.status must be compatible with Order.status
- If Order is "Delivered", all Parcels must be "Delivered"


6.4 MULTIPLICITY RULES
----------------------

One-to-One (1:1):
- Order ←→ Payment (mandatory both sides)
- DeliveryAssignment ←→ Route (mandatory both sides)

One-to-Many (1:*):
- Customer ←→ Order (customer can have many orders)
- Order ←→ Notification (order triggers many notifications)
- Order ←→ StatusHistory (order has many state changes)
- Warehouse ←→ Inventory (warehouse manages many entries)
- Zone ←→ Address (zone covers many addresses)
- Route ←→ Tracking (route has many tracking points)

One-to-Optional (1:0..1):
- Order ←→ PickupAssignment (created after dispatcher action)
- Order ←→ DeliveryAssignment (created after warehouse action)
- Parcel ←→ Inventory (only when in warehouse)

One-to-One-or-More (1:1..*):
- Order ←→ Parcel (order must have at least one parcel)
- Order ←→ ShipmentPicture (order must have at least one picture)

SECTION 8: COMPLETE WORKFLOW THROUGH MODEL
================================================================================

8.1 END-TO-END ORDER LIFECYCLE
-------------------------------

PHASE 1: ORDER CREATION
-----------------------
1. Customer (User) creates new Order
   - Order.status = "Created"
   - Order.orderDate = now

2. Customer uploads ShipmentPicture(s)
   - ShipmentPicture.order → Order
   - At least one picture required

3. Customer enters pickup Address
   - Order.pickup from → Address
   - Address belongs to Zone

4. Customer enters delivery Address
   - Order.deliver to → Address
   - Address belongs to Zone (may be different)

5. System creates Parcel(s)
   - Order contains Parcel(s)
   - Parcel.weight, dimensions, description entered

6. Customer makes Payment
   - Payment created and linked to Order
   - Order paid via Payment
   - Order.status = "Pending Approval" (after payment)


PHASE 2: APPROVAL
-----------------
7. Manager (User) reviews Order
   - Checks Payment.status = "Paid"
   - Views ShipmentPicture(s)
   - Validates against PolicyRule(s)

8. Manager approves Order
   - Manager approves → Order (dependency)
   - Order.status = "Approved"
   - StatusHistory entry created (oldStatus="Pending Approval", newStatus="Approved")

9. System triggers Notification
   - Order triggers → Notification
   - Customer receives → Notification
   - Message: "Order approved"


PHASE 3: PICKUP ASSIGNMENT
---------------------------
10. Manager delegates to Dispatcher
    - Manager delegates to → Dispatcher
    - Order.status = "Assigned to Dispatcher"

11. Dispatcher checks Zone
    - Dispatcher operates in → Zone
    - Verifies Order.pickup Address is in their Zone

12. Dispatcher creates PickupAssignment
    - PickupAssignment links Order and PickupRider
    - Dispatcher creates → PickupAssignment
    - Order has → PickupAssignment (0..1 becomes 1)
    - Order.status = "Pickup Assigned"

13. PickupRider receives assignment
    - PickupAssignment assigned to → PickupRider
    - PickupAssignment.status = "Confirmed"
    - Notification sent to rider


PHASE 4: PICKUP EXECUTION
--------------------------
14. PickupRider navigates to pickup Address
    - Uses Vehicle for transportation
    - PickupRider uses → Vehicle

15. PickupRider collects Parcel(s)
    - Verifies Parcel details
    - PickupAssignment.status = "Completed"
    - Parcel.status = "Collected"
    - Order.status = "In Warehouse"

16. Customer receives Notification
    - Order triggers → Notification
    - Message: "Parcel collected"


PHASE 5: WAREHOUSE PROCESSING
------------------------------
17. WarehouseStaff receives Parcel
    - WarehouseStaff works at → Warehouse
    - Scans Parcel

18. Warehouse creates Inventory entry
    - Warehouse manages → Inventory
    - Parcel stored in → Inventory
    - Inventory.entryDate = now
    - Inventory.binLocation = assigned location
    - Parcel.status = "In Warehouse"

19. WarehouseStaff prepares for delivery
    - Sorts by Zone
    - Quality check


PHASE 6: DELIVERY ASSIGNMENT
-----------------------------
20. WarehouseStaff creates DeliveryAssignment
    - DeliveryAssignment links Order and DeliveryRider
    - Order has → DeliveryAssignment (0..1 becomes 1)
    - DeliveryAssignment assigned to → DeliveryRider

21. System creates Route
    - DeliveryAssignment follows → Route
    - Route.distance calculated
    - Route.estimatedTime calculated
    - Route from Warehouse to delivery Address

22. Inventory updated
    - Inventory.exitDate = now
    - Parcel no longer stored in Inventory (becomes null)

23. Order.status = "Out for Delivery"
    - StatusHistory entry created
    - Notification triggered to Customer


PHASE 7: DELIVERY EXECUTION
----------------------------
24. DeliveryRider follows Route
    - DeliveryRider uses → Vehicle
    - Route tracked by → Tracking (multiple points)

25. Tracking points created continuously
    - Tracking.latitude, longitude, timestamp
    - Tracking.event = "In transit"
    - Customer can view real-time location

26. DeliveryRider completes delivery
    - Parcel.status = "Delivered"
    - Order.status = "Delivered"
    - Route.actualTime = calculated duration
    - DeliveryAssignment.status = "Completed"

27. Final Notification sent
    - Order triggers → Notification
    - Customer receives confirmation
    - Message: "Delivered successfully"


PHASE 8: ANALYTICS
------------------
28. Manager/Admin generates Report
    - Queries completed Orders
    - Analyzes StatusHistory for timelines
    - Reviews Route.actualTime vs estimatedTime
    - Checks DeliveryRider.rating
    - Report created with metrics


8.2 NOTIFICATION WORKFLOW
-------------------------

Trigger Events (Order status changes):
- Order created → Notification to Customer
- Payment confirmed → Notification to Customer
- Order approved → Notification to Customer
- Pickup assigned → Notification to PickupRider
- Parcel collected → Notification to Customer
- Out for delivery → Notification to Customer and DeliveryRider
- Delivered → Notification to Customer

Each trigger:
1. Order triggers → Notification (1:*)
2. Notification → Customer receives (1:1)
3. Notification.channel = SMS, Email, Push
4. Notification.status tracks delivery


8.3 TRACKING WORKFLOW
---------------------

Creation:
- DeliveryAssignment creates Route
- Route tracked by → Tracking points

Updates (continuous):
- DeliveryRider's location polled (every 30 seconds)
- New Tracking entity created
- Tracking.latitude, longitude, timestamp, event
- Linked to Route

Customer View:
- Customer selects Order
- Order has → DeliveryAssignment
- DeliveryAssignment follows → Route
- Route tracked by → Tracking (all points)
- Display on map with breadcrumb trail

9.1 DESIGN PATTERNS USED
-------------------------

COMPOSITE PATTERN:
- Order contains multiple Parcels
- Treated as single unit but composed of parts

OBSERVER PATTERN (IMPLIED):
- Order status changes trigger Notifications
- StatusHistory observes and records changes

STATE PATTERN:
- Order has explicit status attribute
- Valid state transitions enforced
- StatusHistory tracks transitions

FACTORY PATTERN (IMPLIED):
- Different User types created based on role
- Polymorphic user creation

STRATEGY PATTERN (IMPLIED):
- Different payment methods (PaymentMethod attribute)
- Different notification channels (Notification.channel)


9.2 SOLID PRINCIPLES
--------------------

SINGLE RESPONSIBILITY:
✓ Each entity has one clear purpose
  - Payment handles only payment concerns
  - Route handles only routing concerns
  - Clear boundaries

OPEN/CLOSED:
✓ Can extend User with new types without modifying base
✓ Can add new PolicyRules without changing Order

LISKOV SUBSTITUTION:
✓ Any User subclass can substitute User
✓ PickupRider can be treated as User for authentication

INTERFACE SEGREGATION:
✓ Entities have minimal, focused attributes
✓ No "fat" interfaces with unnecessary attributes

DEPENDENCY INVERSION:
✓ High-level Order doesn't depend on low-level implementation
✓ Relationships through abstractions (User, not specific types)


9.3 DOMAIN-DRIVEN DESIGN (DDD) PRINCIPLES
------------------------------------------

UBIQUITOUS LANGUAGE:
✓ Entity names match business terminology
  - "Order" not "DeliveryRequest"
  - "PickupRider" not "CollectionAgent"
  - "Zone" not "GeographicPartition"

BOUNDED CONTEXTS:
✓ Clear contexts visible:
  - User Management Context (User hierarchy)
  - Order Management Context (Order, Payment, Parcel)
  - Logistics Context (Assignment, Route, Warehouse)
  - Communication Context (Notification, Tracking)
  - Analytics Context (Report, StatusHistory)

AGGREGATES:
✓ Order is aggregate root
  - Controls access to Payment, Parcel, ShipmentPicture
  - Maintains consistency boundaries
  
✓ Warehouse is aggregate root
  - Controls Inventory entries

ENTITIES vs VALUE OBJECTS:
✓ Entities (have identity):
  - User, Order, Payment, Parcel, Assignment, Route, etc.
  - Each has unique identifier
  
✓ Value Objects (could be):
  - Address (defined by its attributes, not ID)
  - Could have multiple Addresses with same values


9.4 INFORMATION HIDING
----------------------
✓ Internal IDs not shown in attributes
  - Domain model focuses on concepts
  - Implementation details hidden
  
✓ Complex attributes abstracted
  - "coordinates" not "latitude + longitude" in Address
  - Simplifies understanding

SECTION 13: KEY INSIGHTS AND RATIONALE
================================================================================

13.1 WHY SEPARATE PICKUP AND DELIVERY RIDERS?
----------------------------------------------
Reason: Different skills, vehicles, and workflows
- Pickup: Customer interaction, verification, flexible scheduling
- Delivery: Navigation, time pressure, recipient interaction
- Some riders specialize in one, some do both
- Separate classes allow different attributes and behaviors

13.2 WHY ADDRESS AS SEPARATE ENTITY?
-------------------------------------
Reason: Reusability and consistency
- Same address used for multiple orders (home, office)
- Geocoding done once, reused
- Zone mapping consistent
- Enables address validation and autocomplete

13.3 WHY SPLIT ASSIGNMENTS?
----------------------------
Reason: Different lifecycles and actors
- PickupAssignment: Created by Dispatcher, involves customer interaction
- DeliveryAssignment: Created by WarehouseStaff, involves recipient
- Different confirmation flows
- Clearer semantics in code and database

13.4 WHY VEHICLE SEPARATE FROM RIDER?
--------------------------------------
Reason: Operational flexibility
- Riders can change vehicles (shift changes, maintenance)
- Vehicle can be reassigned to different riders
- Vehicle has its own lifecycle (registration, capacity, maintenance)
- Multiple riders can share vehicles across shifts

13.5 WHY ZONE ENTITY?
---------------------
Reason: Geographic organization
- Dispatchers operate in zones
- Addresses belong to zones
- Routing optimized within zones
- Load balancing across zones
- Enables regional reporting
